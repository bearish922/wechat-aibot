# WeChat AI Bot 产品功能文档与代码 Review

版本：v1.0.0 | 更新日期：2026-05-25

本文基于当前代码库完整 review 生成，面向使用者说明产品已经具备的能力，并记录本次检查发现的 bug、冲突点、隐藏风险和修改建议。

## 1. 产品概览

WeChat AI Bot 是一个运行在 Windows 本机上的微信 AI 助手。它会监听微信消息，把用户发来的文字、图片、语音、文件、视频等内容整理成 AI 能理解的输入，然后交给 Claude Code 或 Codex 回复到微信里。

这个产品现在不只是“问一句答一句”的机器人，而是一个带有多会话、多 AI 后端、角色扮演、本地知识库、多媒体理解、会话恢复和日志记录能力的个人微信 AI 工作台。

当前设计前提是单用户自用：一个微信使用者连接两个 AI 后端（Claude Code 和 Codex）。因此当前 AI、当前角色、当前附件 pending 等状态都按单用户工作流设计，暂不考虑多用户、群聊或多人隔离。

核心适用场景：

- 在微信里直接和 Claude Code 或 Codex 对话。
- 为不同话题创建不同聊天线程，避免上下文混在一起。
- 使用不同角色口吻聊天，例如角色扮演、技术导师、简洁回答等。
- 收到图片、文件、视频时，让 AI 先理解内容再回复。
- 对 BangDream / MyGO 等角色知识进行本地知识库检索，减少角色设定错误。
- 保存每次 AI 运行日志，方便排查问题或恢复旧会话。

## 2. 启动与运行

当前提供三个启动入口：

| 文件 | 作用 |
| --- | --- |
| `start.bat` | 启动微信 AI Bot 主程序。 |
| `rebuild-rag.bat` | 重建本地 BangDream 知识库索引。 |
| `cleanup-media.bat` | 手动清理旧媒体文件（独立脚本，不需要启动 bot）。 |

启动主程序后，如果本地没有有效登录信息，程序会在命令行显示二维码。用户用手机微信扫码确认后，机器人开始监听微信消息。

运行状态会保存在这些文件中：

| 文件或目录 | 用途 |
| --- | --- |
| `weixin-token.json` | 保存微信登录 token、消息同步位置、当前 AI 选择。 |
| `weixin-sessions.json` | 保存 Claude Code / Codex 的会话线程。 |
| `weixin-profiles.json` | 保存角色模板。角色选择不保存在这里。 |
| `config.json` | 保存关键路径、代理、模型、RAG、日志保留天数等可调配置。 |
| `会话恢复指令.txt` | 生成人类可读的 Claude / Codex 会话恢复命令。 |
| `logs/` | 保存每次 AI 调用的详细日志；默认自动清理超过 30 天的日志。 |
| `inbound_media/` | 保存微信收到的图片、文件、语音、视频等媒体；不自动清理，需手动处理。 |
| `rag_vector_store/` | 保存本地知识库向量索引。 |

## 3. 用户能做什么

### 3.1 在微信中直接聊天

用户发送普通文字后，如果前面没有待处理的图片、视频或文件，Bot 会直接把消息送入当前线程，不等待合并。语音消息也会直接进入当前线程。

只有图片、视频、文件这类附件消息会进入短暂合并窗口。当前合并窗口是 30 秒：附件发出后，30 秒内跟上的普通文字会被当作这份附件的补充说明，一起交给 AI；30 秒后再发的文字会独立回复。

如果 30 秒内又发了新的图片、视频或文件，Bot 会立刻结束上一组附件输入，把上一组放入回复队列，然后从新的附件开始重新计算 30 秒。语音和命令不会并入附件说明；它们会让已有附件组先进入队列，再按自己的规则处理。

回复会带有线程前缀，例如：

```text
# CC-S1
回复内容
/
```

其中：

- `CC` 表示 Claude Code。
- `Codex` 表示 Codex。
- `S1` 表示当前线程名。
- 末尾 `/` 表示这一轮回复结束。

### 3.2 切换 AI 后端

用户可以在微信里选择当前使用 Claude Code 还是 Codex。

| 命令 | 作用 |
| --- | --- |
| `/cc` | 切换到 Claude Code。 |
| `/codex` | 切换到 Codex。 |
| `/status` | 查看当前 AI、模型名、线程、角色和会话 ID。 |

Claude Code 和 Codex 拥有各自独立的线程列表。切换 AI 后，用户看到的是对应 AI 自己的会话线程。

### 3.3 多线程会话管理

用户可以在同一个微信入口里维护多个独立会话。适合把不同话题、不同角色、不同任务分开。

| 命令 | 作用 |
| --- | --- |
| `/new [名称]` | 创建新线程。如果不填名称，会自动生成 S2、S3 等名称。 |
| `/switch [序号或名称]` | 切换到指定线程。 |
| `/sessions` | 查看当前 AI 下所有线程。 |
| `/rename [序号|名称] <新名称>` | 重命名指定线程；`/rename <新名称>` 重命名当前线程。 |
| `/close [序号或名称]` | 关闭线程。只剩一个线程时会自动创建新的默认线程。关闭后自动切换到前一个线程并告知当前线程名。 |
| `/cancel` | 取消当前正在运行的任务。 |
| `/cleanup media` | 查看 `inbound_media/` 文件统计（数量和大小）。 |
| `/cleanup media <天数>` | 查看超过 N 天的媒体文件。 |
| `/cleanup media confirm <天数>` | 确认删除超过 N 天的媒体文件。 |

同一个 AI 下，线程名称不能重复。新建线程或重命名线程时，如果名称已经存在，操作会失败并提示换一个名称。

`/rename` 使用空格分隔旧名称和新名称。如果第一个词匹配到已有线程（序号或名称），且后面还有内容，则为 `旧引用 + 新名称`；否则整段都是新名称，作用于当前线程。

线程会被保存到本地文件，程序重启后仍能继续使用。Bot 还会生成 `会话恢复指令.txt`，方便直接从命令行恢复 Claude 或 Codex 的旧会话。

### 3.4 配置文件

`config.json` 用来管理本机相关配置，避免把关键路径和模型写死在代码里。当前包含：

| 配置区域 | 用途 |
| --- | --- |
| `paths` | Claude Code、Codex、RAG 脚本等路径。 |
| `proxy` | 本机代理地址。 |
| `models` | Claude Code 快速模型和 fallback 模型。 |
| `vision` | 图片理解使用的 OpenAI-compatible 视觉模型配置。 |
| `rag` | 本地知识库目录、embedding 模型、召回参数。 |
| `logs` | 日志保留天数。 |

环境变量仍然可以覆盖部分配置，例如 `WEIXIN_CLAUDE_PATH`、`WEIXIN_CODEX_PATH`、`WEIXIN_HTTPS_PROXY`、`WEIXIN_VISION_API_KEY`、`WEIXIN_LOG_RETENTION_DAYS`。

### 3.5 角色模式

Bot 支持把某个线程绑定到指定角色或回答风格。当前配置里已有这些角色模板，每个模板包含完整的角色背景、性格、说话方式、人际关系和任务中的角色表现：

- 默认
- 长崎素世（MyGO!!!!! 贝斯手，包含 CRYCHIC 前史、家庭背景、健康管理习惯等）
- 千早爱音（MyGO!!!!! 节奏吉他手，包含英国留学经历、社交运营等）
- 丸山彩（Pastel*Palettes 主唱，包含假唱事件、努力型偶像内核等）
- 白鹭千圣（Pastel*Palettes 贝斯，包含童星经历、路痴/怕虫等弱点）

用户可以在微信中管理角色：

| 命令 | 作用 |
| --- | --- |
| `/profile` | 查看当前线程角色和所有角色模板。 |
| `/profile <名称>` | 把当前线程绑定到某个角色。 |
| `/profile off` | 当前线程未绑定角色时，恢复默认风格。 |
| `/profile add <名称> | <提示词>` | 添加新角色。 |
| `/profile delete <名称>` | 删除角色模板；如果已有线程绑定该角色，会先提示绑定列表，二次确认后才删除。 |

角色是 session 级别属性，不是全局设置。任何当前状态都可以按三层理解：当前 AI、当前线程、当前线程绑定的角色。

设计上，一个已经绑定角色的线程不能随意切换成另一个角色。默认线程还可以绑定角色；一旦绑定角色，若想换成另一个角色，推荐新建线程。这样可以避免旧上下文残留导致角色混乱。

删除角色不会删除线程。若某个角色仍绑定在一个或多个线程上，Bot 会先列出这些线程；用户在提示时间内再次发送同一条删除命令确认后，角色模板会被删除，这些线程会回退到默认性格。

### 3.6 回复风格控制：长度签 + 场景感知

Bot 通过两层机制控制回复风格：

**长度签（`chooseReplyBudget`）**：每轮根据用户输入类型随机抽取一个回复长度预算。闲聊消息有 65% 概率抽到极短或短档位（≤55 字），正经问题大部分落在正常到较完整档位（80-320 字）。闲聊档位为硬约束（`enforce: true`），超出会被 `constrainCasualReply` 截断；正经问题为软约束，不截断。

**场景感知（`buildStylePrompt`）**：通过 `isInfoSeekingTurn` 判断当前是闲聊还是任务模式，给 AI 不同的风格指令：

- **闲聊模式**：强调自然停顿、接话就停、少用列表式结构和解释腔。颜文字情绪匹配、不重复。
- **任务模式**：允许结构化表达和更长回复，但同时强调保持角色语气——可以用角色式类比、可以吐槽、解决后可以自然收尾，不变成无个性的 AI 助手。

此外，Bot 会记住最近几轮用过的颜文字，避免连续复用同一个。

### 3.7 图片理解

用户发图片时，Bot 会尝试：

1. 下载微信图片到本地。
2. 识别图片格式和尺寸。
3. 调用云端视觉模型生成中文图片描述。
4. 把图片描述和用户问题一起交给 Claude Code 或 Codex。

视觉模型会被要求区分“看清楚的事实”和“不确定推测”，避免凭图片乱猜。

如果没有配置视觉模型 API key，图片仍会保存，但 AI 只能看到“未生成视觉描述”的提示，无法真正看图。

### 3.8 语音、文件和视频处理

当前媒体处理能力如下：

| 类型 | 当前能力 |
| --- | --- |
| 语音 | 保存语音文件；如果微信消息里自带语音转文字，会把文字一起发给 AI。 |
| 文件 | 保存文件；对文本、Markdown、CSV、JSON、PDF、DOCX、PPTX、XLSX 尝试提取文本预览。 |
| 视频 | 保存视频；如果本机有 ffmpeg，会截取首帧，再用视觉模型描述首帧。 |

注意：旧版 Office 格式如 `.doc`、`.xls`、`.ppt` 虽然能识别文件类型，但当前不会直接提取正文预览。

### 3.9 本地知识库检索

当启用角色模式，并且用户问题不是图片/文件/视频类输入时，Bot 会尝试查询本地 BangDream 知识库。对于已绑定非默认角色的线程，查询时始终以角色名作为前缀进行锚定（如 `"长崎素世 用户消息"`），大幅提高日常对话中角色相关知识的命中率。纯问候语（"早上好""晚安""哈哈"等）仍会跳过 RAG。

当前知识库配置：

| 项目 | 当前值 |
| --- | --- |
| 知识库目录 | `D:\Desktop\Obsidian\entertainment\Bangdream-Knowledge` |
| 向量模型 | `BAAI/bge-small-zh-v1.5` |
| 当前索引块数 | 138 |
| 默认召回数量 | 3 |

知识库的作用是为角色设定、人物关系、经历、学校、乐队等信息提供本地背景，减少 AI 编造设定。

### 3.10 日志和排查

每次 AI 运行会生成两类日志：

- `.jsonl`：保存原始事件流，方便精确排查。
- `.txt`：保存人类可读摘要，包括用户输入、工具调用、结果、错误等。

媒体类输入还会写入 `logs/inbound-media.jsonl`，记录原始媒体消息元数据。

日志会自动清理，默认保留 30 天。可以通过环境变量 `WEIXIN_LOG_RETENTION_DAYS` 调整保留天数；设置为 0 或负数时关闭自动清理。媒体文件不会自动删除，需要用户手动清理 `inbound_media/`。

这些日志对排查很有用，但也可能包含聊天内容、文件路径、图片信息和用户 ID，因此需要注意隐私和清理。

## 4. 当前产品边界

以下能力目前还不是完整产品化状态：

- 没有图形化管理界面，主要通过微信命令操作。
- 没有正式测试套件，当前只有语法检查和人工验证。
- 媒体文件没有自动清理策略（已有手动清理命令和脚本，但不会自动清理）。
- 当前按单用户自用设计，不承诺多人或群聊使用。
- 文件解析只做预览，不是完整文档阅读系统。

## 5. 本次 Review 结论

整体来看，这个产品已经从“简单微信机器人”演化成了一个功能很丰富的个人 AI 聊天工作台。主流程能跑通，代码也有不少实用保护：单实例锁、会话持久化、Claude/Codex 会话恢复、RAG 超时、媒体下载失败兜底、取消任务、日志留存等。

当前最大的工程问题是：功能增长都堆在一个 2000 多行的 `bot.mjs` 里，状态边界开始变复杂。因为产品明确是单用户自用，`activeAI` 这类全局状态本身可以接受；更需要关注的是两个 AI 后端、多个线程、session 级角色绑定和附件 pending 之间的状态一致性。

## 6. 发现的问题和修改建议

### P1：附件消息合并的状态边界（已解决）

位置：`bot.mjs` 的 `enqueueUserBody`、`flushPendingInput`、`queueTurn`、`activeSession` 附近。

原问题：pending 状态与线程/AI 的边界可能在某些边缘场景下错位。

已完成修改：
- pending 保存 `ai + sessionId`（已完成）。
- `/cancel` 清理 pending 消息（已完成）。
- `/close` 目标线程时检测并清除该线程的附件 pending（2026-05-25 完成）。

### P2：微信 API 返回值没有统一校验（未处理）

位置：`apiPost`、`sendMessage`、`mainLoop`。

现象：

- `apiPost` 直接解析 JSON，没有统一检查 HTTP 状态和业务错误。
- `sendMessage` 调用后没有检查是否真的发送成功。

用户影响：

- 某些发送失败可能只出现在日志里，用户侧看不到原因。
- token、限流、接口变更等问题不容易定位。

建议：

- 为微信 API 增加统一返回校验。
- `sendMessage` 失败时记录目标、错误码、短文本摘要。
- 对可恢复错误增加重试，对不可恢复错误给用户提示。

### P3：文件预览能力容易被用户误解为完整文件阅读

位置：`extractFileTextPreview`、`runPythonExtractor`。

现象：

- PDF 只读前 8 页。
- PPTX 只读前 20 张 slide。
- XLSX 只读前 8 个 sheet，且只是抽取部分单元格文本。
- 解析失败时静默返回空。

用户影响：

- 用户可能以为 AI 已经完整阅读文件，实际只是看了预览。

建议：

- 在发给 AI 的提示里明确“这是文件预览，不代表完整文件”。
- 对解析失败给出更明确的用户可见提示。

### P3：缺少正式测试和结构化回归验证

位置：项目整体。

现象：

- `package.json` 里没有可用测试。
- 当前可以做 `node --check bot.mjs` 和 `python -m py_compile rag.py`，但没有覆盖实际行为。

用户影响：

- 功能越多，改动一个命令时越容易影响另一个命令。
- 会话、角色、pending、媒体等状态组合很难靠人工完整验证。

建议：

- 先补最小测试，不必一次做大：
  - 命令解析测试。
  - pending 消息路由测试。
  - 角色绑定/删除测试。
  - 文本拆分测试。
  - RAG skip/触发测试。
- 把纯逻辑函数从 `bot.mjs` 拆出来，方便测试。
