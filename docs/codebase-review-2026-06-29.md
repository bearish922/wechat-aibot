## 二、先说总判断

当前项目已经是一套“长期角色连续性系统”，不只是微信消息转发器。它同时维护：

- 用户看得到的聊天历史；
- 角色此刻在什么地方、做什么；
- 长期记忆和情景记忆；
- 跨天生活线与工作日程；
- 主动消息；
- CC、Codex、Direct API 三种后端；
- 普通千圣、梦中千圣以及更多角色的隔离。

最核心的单 Actor 回复链已经相对稳定。近期记录里，普通单 Actor 调用 30 次成功 29 次；4 次格式错误后的自动重试全部成功。真正较弱的是“回复之后的后台整理”：连续性更新成功 17/28，日程提取成功 26/39，情景记忆生成成功 3/5。也就是说，用户当下经常能收到正常回复，但角色的内部账本不一定同步跟上。

我的总体判断：

1. 不建议再次大拆架构。当前“一个 Actor 负责内心和可见回复，辅助任务在回复后独立运行”的方向是合理的。
2. 下一阶段应先修“对话是否真的发生”“取消是否真的生效”“状态是否真的落下”这三类可靠性问题。
3. 角色自然度目前更受后台状态漏记、错误日程和配置不一致影响，而不只是提示词文风。
4. 梦中千圣的运行隔离大体正确；更大的结构风险反而是“多微信用户共用同一角色世界”。

## 三、第一部分：已经直接修复的普通代码问题

以下修改不需要判断“千圣应该怎么说话”，所以已经应用。

### 1. 数据保存与恢复

- 会话文件、角色世界、聊天数据库和微信 token 改善了原子保存或备份恢复，减少写到一半导致整个文件损坏的风险。
- 主会话文件损坏时会尝试读取备份；空或非法的当前会话会自动回到一个有效会话。
- Memory 和 World Memory 现在允许保存空内容，真正实现“清空”，不再因为空字符串而保留旧内容。
- 聊天数据库保存时不再直接覆盖在线文件，而是先生成临时文件，再替换并保留备份。
- 天气接口某一个城市失败时，不再把另一个城市已经成功取得的新数据也一起丢掉。

### 2. CC、Codex 和 Direct API

- Direct API 的 429、500、502、503、504 重试判断已修正。
- 流式响应的超时现在覆盖完整正文读取，不再只覆盖“等到响应头”。
- 工具调用在一次临时失败后能够继续完成后续轮次。
- 支持某些服务商把 token 用量单独放在最后一个空内容数据包里的情况。
- Codex 会话用量缓存现在会在日志文件变大时刷新，不会长期停在旧值或空值。
- 用磁盘日志补全 Codex 上下文窗口时，不再反过来用较旧数据覆盖内存里的最新 token 数。
- 后端适配器避免把 Claude 的增量文本和最终完整文本重复拼接。
- 启动时 Direct API 历史只读取一次，不再为每个用户重复扫描整库。
- Claude 子进程的会话显示名会过滤命令行特殊字符，降低意外执行额外命令的风险。
- 清理 Claude 旧会话时，只清理由本项目明确创建的隐藏角色会话，不再碰同一目录下的其他 Claude 工作记录。

### 3. 微信发送、回复切分和媒体

- 同一条微信消息重试发送时复用同一个 `client_id`，降低网络抖动造成重复消息的概率。
- 微信接口现在检查 HTTP 状态，不再把 4xx/5xx 页面误当成正常 JSON。
- 极小的回复分片上限不再可能让切分循环卡死。
- 音频不是 Silk 格式时，会真正通过 ffmpeg 转码，不再只是把文件改成 `.wav` 后缀。
- Whisper 成功后会清理临时文件。
- 媒体处理超时时会终止整棵子进程，避免 ffmpeg 或识别程序留在后台。
- 下载前会先检查服务器提供的文件大小。
- 媒体日志不再记录微信上下文密钥、AES 密钥和带签名下载地址。

### 4. RAG 与模块结构

- 并发 RAG 查询不再共用同一份临时文件，避免两个对话互相覆盖。
- RAG 脚本缺失、无结果和异常时统一返回空文本，不再让后续读取长度报错。
- 向量维度改为从实际嵌入模型结果读取，不再硬编码 512。
- 删除一层重复的关系过滤。
- 清除了运行模块之间的循环引用；当前 37 个运行模块没有依赖环。

### 5. GUI

- Memory 编辑、预览、切角色时会保留或提醒未保存内容；失败不再显示成功。
- History 的非法页码不会导致异常；编辑或删除 Direct API 历史后会同步保存 API 上下文。
- 角色、提示词等重要 JSON 写入改为临时文件替换。
- 修复了一个隐私问题：保存主提示词时，过去可能把本地私密覆盖项合并进公开主文件；现在公开文件只来自公开内容。
- Prompt 延迟自动保存会记住触发保存时的角色，不会在用户快速切角色后把内容写给另一个角色。
- Prompt 页原本写了很多步骤说明却没有真正渲染，现在会显示。
- 特殊日期编辑会提交最后一次正在输入的内容。
- Config 增加了主模型、上下文上限、语音、更多 RAG 参数和角色专用 Claude 配置映射。
- Status 正确显示 CC、Codex、API 三类会话数量与当前后端。
- Codex 的隐藏会话恢复命令已修正。
- 移除了外部 Markdown CSS CDN，管理页离线时不再依赖第三方网站，也不会无端向第三方发请求。
- 删除未调用的弹窗、旧世界状态组件、旧管道渲染和多余 CSS。
- 我用本地模拟数据实际打开了 Status、Prompts、History、Proactive、Memory、Config 六个页签；均能渲染，浏览器无脚本错误。

### 6. 配置、脚本、安装和死代码

- 删除两个完全没有入口的旧运行模块：`api-session.mjs`、`gui-context.mjs`。
- 删除一次性人工 reset 脚本和一份约 50 KB 的真实情景 Prompt 快照。
- 删除已经没有读取方的 `hiddenWorldMaxPendingIntents` 配置。
- 多个评测脚本从旧 JSON 历史改为读取当前 SQLite 历史，并修正已经移动的角色资料路径。
- `init-memory.mjs` 改为可传角色和用户，不再写死个人信息。
- CI 和发布流程会先安装锁定依赖再测试；安装脚本使用 `npm ci` 和 `python -m pip`。
- 示例配置删除已失效字段，并把 WhisperX Python 示例从一句说明文字改为可执行的 `python`。
- `launch.bat` 不再因为 18720 端口被占用就杀掉任意程序；只会重启明确属于本项目的 `app/bot.mjs`。
- 删除被版本库追踪的 `.claude/settings.local.json`，并加入忽略规则，避免个人用户名和本机权限配置进入发布包。
- README 和版本号已与当前 GUI、当前版本对齐。

## 四、当前项目实际怎样运行

用户消息进入后，大致经过这条链：

1. 微信接收与去重。
2. 把短时间内连续发送的文字和附件合并。
3. 确定当前后端、会话和角色。
4. 如果是角色聊天，先调用单 Actor，一次生成“角色内心叙事”和“用户可见回复”。
5. 把可见回复分段发到微信。
6. 发送成功后写聊天历史。
7. 再异步做连续性更新、日程候选提取、主动消息候选等后台整理。

这套顺序有一个好的核心：后台整理失败不会阻塞用户收到回复。但它也产生了当前最重要的缺口——回复和内部状态并不是一个整体事务，后者失败后没有可靠补账。

## 五、第二部分：业务与用户体验问题

本章所有内容都只是报告，没有修改。

### A 级：建议最先确认

#### A1. 发送失败时会制造“幽灵对话”

当模型已经生成一部分内容、但微信发送失败时，异常分支仍可能把该内容和内心叙事写入历史。之后角色会把它当作自己已经说过的话，但用户实际没有收到。

影响：

- 用户和角色对“上一句话”拥有不同记忆；
- 重试时可能被旧的未送达内容影响；
- 这是长期角色连续性里比偶尔文风不自然更严重的问题。

建议方向：只有微信确认发送成功后，才让这轮进入正式历史和角色世界。失败内容可单独进入诊断日志，但不能成为角色记忆。

这也与过去项目约定“失败轮次不应推进或污染正常上下文”冲突。当前代码应作为事实来源：这个约定目前只在部分路径成立，没有完整兑现。

#### A2. `/cancel` 的承诺比实际能力大

当前取消主要能停止主后端的流式回复，却不能可靠停止已经发出的单 Actor、状态更新、日程审批等结构化调用。用户收到“已取消”后，后台调用仍可能完成，甚至稍后继续发回复或改状态。

建议方向：取消应该绑定到整轮消息，而不是只绑定到某一个子进程。所有后续步骤在提交结果前都检查同一个“本轮已取消”标记。

#### A3. 回复后的内部记账成功率不足

近期 182 次辅助调用中：

| 辅助步骤 | 成功/总数 | 直观含义 |
|---|---:|---|
| 单 Actor | 29/30 | 主回复稳定 |
| 单 Actor 格式重试 | 4/4 | 重试有效 |
| 连续性更新 | 17/28 | 约四成没有完成 |
| 日程提取 | 26/39 | 约三分之一没有完成 |
| 主动消息二次判断 | 20/20 | 稳定 |
| 每日分享种子 | 24/25 | 稳定 |
| 时间推进 | 24/25 | 稳定 |
| 工作事件生成 | 4/5 | 样本少，基本可用 |
| 情景记忆 | 3/5 | 样本少，但失败比例高 |

这些失败多数不影响当场回复，所以不容易被察觉，却会逐渐表现成：忘记刚确认的事、日程没有落下、位置或活动更新慢。

建议方向：把辅助步骤放进一个可重试的小队列，记录“这轮哪些账还没记成功”，而不是失败后只写日志。

### B 级：会直接影响角色世界是否可信

#### B1. 主动消息看不到真正的生活线

生成主动消息时，代码把普通聊天会话对象传给了读取生活线的函数，而生活线实际存在角色世界对象中。因此提示词里宣称会参考活跃日程，实际很可能一直是空的。

结果是主动消息更依赖临时灵感，不能稳定联系到角色最近在忙什么。建议修正传入对象，并增加一条“有生活线时必须出现在二次判断输入中”的测试。

#### B2. 日程审批缺少最近对话，失败候选还会反复出现

日程审批模板支持“最近可见对话”，但调用方没有传入。审批模型只能看候选摘要和已有日程，无法核对用户刚才到底确认了什么。

同时，模型返回格式不合格时，函数会报告“处理过”，却不清除候选；同一批坏候选会在下一轮再次审批。

建议方向：审批时带上最近几轮对话；明确区分“审批成功”“主动跳过”“格式失败”，格式失败有限重试后进入人工可见的失败区。

#### B3. 工作事件的冲突检查没有使用同一套字段

生成器会得到 `duration_hours`，但写入生活线时该字段被标准化过程丢弃。后续“当天已经工作几小时”的判断因此缺少关键数据。

硬冲突检查主要读取 `timeSlots`，但新生成的工作事件主要保存为 `timeStart/timeEnd`。这样两个看似都有具体时间的工作仍可能重叠。事件最小间隔也主要靠提示词要求，没有完整的代码检查。

建议方向：先统一日程数据格式，再做纯代码的时间重叠和每日工时检查；提示词只负责生成内容，不负责最后守门。

#### B4. 东京角色有些“每日逻辑”仍按上海电脑时间

工作事件的明显时区错误已作为普通 bug 修复，但时间推进、每日分享、主动消息的“今天”和“时段”仍有路径直接使用电脑本地时间。上海和东京只差一小时，平时不显眼，午夜附近会跨到不同日期。

建议方向：角色世界内所有“今天、星期、早晚、每日计数”统一使用角色时区；微信日志时间可继续使用北京时间。

#### B5. 已结束生活线会在加载时消失

标准化过程只保留 active 项目，closed 或 expired 的生活线会被丢弃；GUI 却设计了过期分组。这意味着界面承诺能看历史，数据层实际没有完整保留。

角色也因此无法知道“上个月的拍摄已经结束”，只能知道它不在当前列表。建议把当前与归档分开保存，而不是加载时删除。

#### B6. 多微信用户会共用同一个角色世界

可见聊天历史按“微信用户 + 角色”隔离；但角色位置、生活线、主动消息、情景记忆、长期记忆主要只按角色名保存。

对目前基本只有一个用户的使用方式，这不会立刻出问题。但一旦第二个微信用户与白鹭千圣聊天：

- 两人会影响同一个角色当前地点和日程；
- 一个用户触发的主动消息候选可能被另一个用户看到或消费；
- 长期用户记忆不能真正区分两个人。

建议先确认产品定位：

- 如果永远是单人私用，明确写成限制，保持结构简单；
- 如果允许多人使用，角色世界至少应按“用户 + 角色”隔离，公共设定再单独共享。

### C 级：功能可用，但存在明显错位

#### C1. Native 图片模式目前名义上存在，能力上并不成立

README 和 Config 提供“Native backend”视觉模式，但该模式不会走外部视觉识别，同时给主模型的文字又明确说不要读取本地图片路径。结果很可能是模型只知道“收到一张图片”，却看不到图片。

建议方向：要么真正把图片作为后端支持的附件发送；要么暂时移除 Native 选项和 README 承诺，只保留外部视觉 API。

#### C2. Prompt 页的 RAG 参数和生产检索参数不是同一套

Prompt 页可以编辑 Top-K、最低分、最大字符数、超时；但生产 `rag.py` 主要读取 `data/config.json`。用户在 Prompt 页改了数值，可能以为已经改变真实检索，实际没有。

角色选择器也会造成误解：关键词判断读取的是全局提示词，不是所选角色的独立设置。

建议方向：只保留一个配置来源，并在 GUI 上明确标注“全局”还是“当前角色”。

#### C3. RAG 建库和检索还有四个结构性弱点

1. 每次查询都会新启动 Python，并重新加载嵌入模型，速度和内存开销较大。
2. 重建向量库时先删除旧库，再建新库；中途失败会让可用旧库也消失。
3. 启动时只检查元数据是否存在，不检查知识文件是否比索引更新。
4. 全部 203 份资料都没有 frontmatter 元数据，历史时期、当前有效性只能靠路径和文本猜测。

另一个内容层事实是：事件目录目前只有 1 份索引文档。角色、关系资料很丰富，但事件时间线相对稀薄。

建议方向：先做“索引新鲜度 + 原子重建”，再考虑常驻检索服务；给最容易过期或互相冲突的资料补少量明确元数据，不必一次给 203 份全部加。

#### C4. 媒体处理仍有产品限制

- 相同媒体的内部文本包含随机本地路径，微信重投时可能绕过内容去重。
- 没有自动保留期限，媒体只能靠人工运行清理脚本。
- PDF、表格等只取有限页数或有限内容，用户未必知道后半部分没有被阅读。
- 多个附件顺序处理，整轮等待时间会叠加。
- 即使图片说明文字明确询问角色资料，只要带附件，当前轮也会跳过 RAG。

建议让回复显式说明“只读取了前 N 页/部分工作表”，并给媒体文件设置默认保留期。

#### C5. 情景记忆的实际注入范围与页面说明不一致

GUI 和文档说情景记忆只在 reset 后第一轮注入；代码实际会在 Actor 前 15 次调用继续注入，Direct API 更是每轮注入。

这不一定是坏设计，但属于需要明确的业务选择。长时间重复注入能提高连续性，也会让旧摘要压住新对话、增加上下文。建议决定是“只首轮”“前几轮衰减”还是“直到新状态足够稳定”，然后让代码和页面一致。

#### C6. 会话和角色绑定容易让人误解

- 输入会话名的部分文字时，多个匹配会直接取第一个，没有歧义提示。
- 把已有会话改绑到另一个角色时，旧的可见上下文仍可能影响新角色。
- CC、Codex、API 各自有当前会话，但切后端是全局动作，切换后当前角色可能也随之变化。
- 关闭一个非当前会话时，会清理用户级待处理附件，可能影响当前会话正在合并的输入。

建议在改绑角色前强制选择“新建干净会话”或“保留旧历史”；歧义匹配不要自动猜。

#### C7. Direct API 与 CC/Codex 不是完全等价的角色后端

Direct API 的上下文由本项目保存；CC/Codex 还有各自的原生会话。API 消息在模型成功后就进入内存上下文，而微信发送可能随后失败，因此更容易出现“模型记得但用户没看到”。

API 自带的 `web_fetch` 还能访问任意 URL，目前没有限制本机地址或内网地址。这在单人本机使用时风险较低，但如果以后允许他人发消息，可能被利用去读取本机或局域网服务。

建议把 Direct API 明确标成“兼容模式”还是“正式等价后端”；若正式支持，应补发送成功提交和 URL 访问边界。

#### C8. 历史库在数据变大后会慢

当前每新增一条消息，都由 `sql.js` 把整个数据库导出并写回文件。历史搜索也有路径会先取出大量匹配项，再在内存分页。数据量小时没有问题，长期使用后保存和搜索会越来越重。

建议设一个实际规模门槛，例如数据库达到 100 MB 或保存耗时超过 300 ms 时，再迁移到原生 SQLite；现在不必为未来过早重写。

#### C9. GUI 仍更像维护面板，不是普通用户控制台

- GUI 展示“所有用户中最近活跃的会话”，会话表却不显示用户 ID，多用户时难判断是谁。
- 不能在 GUI 切换 CC/Codex/API。
- Busy 和 Queue 不持续刷新，页面可能显示旧状态。
- Prompt 自动保存缺少撤销、变更预览和“一键恢复”。
- Proactive 页面只能看，不能取消待发送意图或处理失败项。
- History 编辑的是本地 SQLite 可见历史，也就是项目后续注入的 visible context / visible reply history。它不能手术式改写 CC/Codex 已经写入的原生会话线程；如果原生线程里也需要丢弃旧内容，只能 reset 对应后端线程。

建议优先增加“只读安全模式、修改前差异、恢复上一个版本”，而不是继续堆更多字段。

## 六、你特别提到的三类架构

### 1. CC / Codex / Direct API 切换

做得好的地方：

- 三类后端拥有各自的用户会话。
- CC 和 Codex 的 Actor 线程分开，切换后不会误用对方 SID。
- 角色世界内容共享，因此换模型不需要从零认识角色。
- Codex 上下文压力已经按“当前压力”显示，不再把累计计费 token 当成当前上下文。

需要确认的地方：

- 切换后端是全局的，不是按用户或会话。
- 同一个角色在三种模型上的行为能力并不完全一致。
- GUI 只能看当前后端，切换仍要在微信里发命令。
- API 和 CLI 后端的“成功提交点”不同，应统一。

结论：保留三后端架构是有价值的；下一步应统一用户可感知的成功、取消、重试语义，而不是强行让底层实现完全一样。

### 2. 普通千圣与梦中千圣

当前隔离情况：

- 不同 profile；
- 不同角色世界；
- 不同 Actor 会话；
- 不同长期记忆和情景记忆；
- 不同可见历史；
- 梦中千圣关闭主动消息、生活线、天气和工作事件；
- 梦中千圣跳过普通连续性更新，并允许把 `inner_scenelet` 作为可见回复。

这说明运行隔离整体是正确的。

仍需注意：

- 两者 RAG 都映射到同一套千圣公共资料。这适合共享公开事实，但不适合存放只属于梦境叙事的私密经历。
- 梦中千圣的角色名和基础配置出现在被追踪的主配置中，而部分注释仍声称敏感角色只存在本地覆盖。代码事实与这条隐私承诺冲突。
- 本地角色提示词文件损坏时会静默回退到主提示词，可能让梦中千圣在用户不知情时使用普通角色规则。

建议：公共人物事实继续共享；梦境专属经历单独存放；本地提示词加载失败应在 GUI 和启动日志中明确报警，而不是静默降级。

### 3. 多角色架构

做得好的地方：

- 每个角色可以拥有独立提示词套件、世界、记忆、历史和 Actor 线程。
- 新角色默认走完整单 Actor 流程，不会自动继承白鹭千圣的私有文风。
- 角色专用 Claude 配置目录为服务商隔离留出了入口。

当前限制：

- 除白鹭千圣和梦中千圣外，多数角色只有通用默认规则，没有经过角色级调校。
- 普通新角色默认开启主动消息、生活线和天气，这未必适合所有角色。
- 工作事件只对白鹭千圣有现实意义。
- 一些旧字段仍留在角色配置或历史实验脚本中，但生产代码已经不读取，容易让人误以为改它们有效。

建议：新建角色时使用“能力模板”选择，而不是一律全开，例如“现实生活角色”“纯叙事角色”“梦境角色”“普通助手”。

## 十、过去记忆与当前代码冲突

1. 过去约定“失败轮次不推进、不污染上下文”。当前普通成功路径遵守，但发送失败的异常路径仍可能写历史，属于未完整实现。
2. 过去强调 retry 不应新开 session。当前单 Actor 格式重试仍在原 SID 内完成，这一点与约定一致。
3. 过去把情景记忆描述为 reset 后首轮注入。当前代码是前 15 个 Actor 调用，API 每轮注入，已经发生偏离。
4. 过去把梦中角色的敏感配置视为本地隔离。当前仓库主配置已经追踪梦中千圣的部分存在信息，隐私边界发生变化。
5. 过去讨论过拆分 Actor、`scene_state` 和 Finalizer。当前生产代码已经选择单 Actor；旧实验脚本和数据库旧列仍保留，但不代表现行架构。

## 十一、建议你按这个顺序决定

### 第一批：可靠性，不涉及角色审美

1. 只有微信发送成功才提交历史和 API 上下文。
2. 让 `/cancel` 覆盖整轮 Actor 和所有辅助调用。
3. 给连续性、日程、情景记忆增加失败重试队列。
4. 统一工作日程字段，并用代码检查重叠、间隔和每日工时。

### 第二批：需要你确认产品含义

5. 项目是否永远只服务一个微信用户；若是，就明确单用户边界。
6. 情景记忆到底只注入首轮、前几轮，还是每轮。
7. Direct API 是正式等价后端，还是临时兼容后端。
8. Native 图片模式是补齐真实能力，还是暂时移除。
